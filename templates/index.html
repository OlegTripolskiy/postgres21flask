Целостность БД.

Целостность базы данных (database integrity) — соответствие информации, имеющейся в базе данных, внутренней логике, структуре и всем явно заданным правилам базы данных.

Например:
  1. Не должно быть явных противоречий между информацией, об объектах хранящейся в базе данных и самими объектами.
    - Вес запчастей, продуктов и т.д. (физических объектов реального мира) не может быть отрицательным.
    - Цена товара не может быть отрицательной.
    - У номера кредитной карты, например Visa, должно быть 16 цифр.
    - Почтовый код города должен соответствовать определенному формату, принятому в стране и у разных городов не должно быть одинакового кода.
  2. Информация должна соответствовать внутренней логике.
    - Если в БД указана минимальная цена продажи товара, то товар не должен продаваться ниже этой цены.
    - У конкретного товара в базе данных не должно быть двух и более наименований.
    - Ученик в школе может числиться только в одном классе.
    - Покупатель может приобрести много товаров, но товар не может купить покупателя.
  3. Данные должны соответствовать заданной структуре.
    - У любого сотрудника предприятия должен быть паспорт и, причем только один.
    - Сотрудник предприятия может числиться только в одном подразделении (или в нескольких).
    - Один и тот же вид товара может храниться на разных складах.

Таким образом, целостность означает правдоподобность (непротиворечивость) информации хранящейся в БД. Целостность означает, что в базе данных нет заведомо невероятных и невозможных значений. Для обеспечения целостности БД требуется обладать знаниями о соответствии информации, хранящейся в БД и заданным для нее правилам.

Целостность БД НЕ гарантирует достоверности (истинности) содержащейся в ней информации. В базу данных пользователи могут ввести недостоверную информацию.

Достоверность (истинность) означает, что информация, хранящаяся в БД, соответствует реальным фактам.
Не следует путать целостность и истинность базы данных – это разные понятия и могут реализовываться по-разному.

Ограничения целостности БД.

В СУБД имеются встроенные средства, которые могут существенно повысить целостность базы данных. Эти средства называются ограничениями целостности.

Ограничения целостности – специальные средства в CУБД, позволяющие налагать некоторые ограничения на информацию в базе данных, позволяющие этой информации быть правдоподобной. Рассмотрим еще несколько примеров правил, которые требуют введения ограничений.

Например:
  1. Возраст биологических родителей должен быть больше (на какое-то количество лет) возраста их ребенка.
  2. Количество символов в почтовом индексе каждого города в стране должно быть одинаковым.
  3. Количество дней в неделе не может быть больше 7.
  4. Каждый сотрудник на предприятии должен быть уникально идентифицирован. Например, иметь уникальный табельный номер, чтобы исключить начисление зарплаты одному сотруднику, а выплату денег другому.

Фактически, любое ограничение является следствием каких-то свойств объектов и их взаимосвязей. База данных не понимает «смысла» данных, которые в ней хранятся. Для того чтобы информация была хотя бы правдоподобной, в базу данных и встраивают средства, позволяющие это обеспечить. В следующем параграфе рассмотрим, на простом примере, работу ограничения целостности, встроенного в СУБД.

Простое ограничение целостности - типы данных.

Типы данных сами по себе уже являются ограничениями целостности. При создании таблицы, в которой будут храниться данные необходимо для каждого столбца определить тип данных, который в этом столбце будут храниться. Если же мы попытаемся ввести данные другого типа, то СУБД не позволит нам этого сделать и выдаст предупреждение.

Проверим работу такого ограничения. Для этого создадим вот такую таблицу.

pid pname pburn
1 Иванов 2010-03-25
2 Петров 2011-08-14
3 Сидоров 2010-06-18

В ней будем хранить информацию об учениках школы. Сначала введем правильные данные, а затем попытаемся ввести в столбец данные другого типа.
Все имена столбцов будут начинаться с английской буквы 'p' (от слова pupils).
Название таблицы – pupils.

  1. Создаем таблицу. 
    CREATE TABLE pupils (  
    pid integer,    /* номер, тип данных: целое число */
    pname text,     /* фамилия, тип данных: текст */
    pburn date	    /* дата рождения, тип данных: дата */
);
  2. Заносим в таблицу данные. 
    INSERT INTO pupils  VALUES 
    (1, 'Иванов',  '2010-03-25'),
    (2, 'Петров',  '2011-08-14'),
    (3, 'Сидоров', '2010-06-18')
;
  3. Смотрим, содержимое таблицы. 
    SELECT * FROM pupils;
-- ответ сервера
pid |  pname  |    pburn    
-----+---------+------------
   1 | Иванов  | 2010-03-25
   2 | Петров  | 2011-08-14
   3 | Сидоров | 2010-06-18
(3 rows)
    Все в порядке, данные в таблицу занеслись корректно.
  4. Проверяем, что задавая столбцам тип данных, мы уже ограничиваем вносимые данные. Пробуем ввести текст вместо даты. 
    INSERT INTO pupils (pid, pname, pburn) VALUES (
    4,        
    'Добров',   
    '18 июля 2012'  /* Дата, записанная текстом */
);

-- Получаем ошибку.  
ERROR:  invalid input syntax for type date: "18 июля 2012"
LINE 4:     '18 июля 2012'  /* Дата, но записанная текстом */
            ^
    Ограничение целостности сработало. СУБД предотвратила операцию и выдала нам предупреждение об ошибке и ее расположении.

Этот простой пример приведен только для того, чтобы рассмотреть само понятие ограничения целостности. Надо отметить, что в этом примере не все будет работать строго. При вводе значений в таблицу, СУБД постарается преобразовать получаемое значение к типу данных столбца.

Например:
  1. В столбец pid (тип данных: целое число) при вводе ТЕКСТА похожего на число, СУБД постарается преобразовать его в ЧИСЛО.
  2. В столбец pname (тип данных: текст) при вводе ЧИСЛА, СУБД постарается преобразовать его в ТЕКСТ.
Пример. Попробуем ввести в таблицу некорректные данные.
  1. Вводим данные. 
    INSERT INTO pupils (pid, pname, pburn) VALUES (
    '4',         /* Текст, похожий на число */
     125.00,     /* Число вместо текста */
    '2012-07-18'
);
    Никаких сообщений об ошибке мы не получили. Похоже, что данные благополучно записались в БД.
  2. Смотрим записи в таблице 
    SELECT * FROM pupils;
-- Получили ответ.
 pid |  pname  |   pburn    
-----+---------+------------
   1 | Иванов  | 2010-03-25
   2 | Петров  | 2011-08-14
   3 | Сидоров | 2010-06-18
   4 | 125.00  | 2012-07-18
(4 rows)

СУБД преобразовала информацию в типы данных для соответствующих столбцов и записала их в таблицу. Внешне, в столбец "pname" записалось число, хотя СУБД считает это текстом и даже попыталась нам помочь, преобразовав один тип данных в другой.

Термины.

Для успешного рассмотрения различных ограничений целостности, надо разобраться с некоторыми терминами. В соответствии с этими терминами будут и называться ограничения целостности.

  1. Сущность (entity) – это объект реального мира, который может быть четко отделен от других объектов.

    Набор (множество, тип) сущностей – группа однородных сущностей (группа объектов одного типа).

    В разных ситуациях и источниках могут использовать разные термины для того, чтобы обозначить группу однотипных объектов.

    Рассмотрим на примере.
    1. Сущность - школьник Петров Коля. Тип сущности - школьники.
    2. Сущность – одноклассник Петров Коля. Группа сущностей – одноклассники.
    3. Сущность – звезда Альдебаран. Множество сущностей – звезды.

    Даже в этом простом примере, можно заметить, что мы часто используем в названии сущности еще и название группы, в которую мы эту сущность включаем. Петров Коля – это реальный человек. Но группы, в которую мы его включаем, уже говорит об информации, которая нас интересует.


    Различимость сущностей - является очень важным понятием.

    Каждая сущность должна быть строго идентифицирована, хотя бы в пределах своей группы. В дальнейшем, даже будет говориться об уникальности каждой конкретной сущности.

    Например, автозавод выпускает множество одинаковых автомобилей. И у определенной модели автомобиля может быть одинаковым все (цвет, набор дополнительных устройств и т.д.). Но каждый автомобиль должен быть идентифицирован и для этого ему на кузове выбивают специальный идентификационный номер.

    Для этой же цели, чтобы точно идентифицировать любой автомобиль, зарегистрированный в стране, государственными органами выдается регистрационный номер, без которого эксплуатация автомобиля запрещена.

    Еще пример из торговли товаром для строительства - песком. Отдельные песчинки не являются сущностями – мы не сможем их как-то идентифицировать. Но песок может быть речной, карьерный, морской. Их идентифицируют по той местности, где они добываются т.к. это позволяет обозначить их особенности и даже отличить их визуально.

    Различимости можно добиться, идентифицируя товар с помощью имени фирмы, которая его поставляет. Например, качество муки определенного сорта определяется стандартами и должно быть одинаковым. Но многие люди предпочитают покупать муку от определенного производителя. В торговле медицинскими лекарствами фирма-производитель зачастую является вообще определяющим. От этого зависит и цена и качество лекарства.

    ЗАМЕЧАНИЕ. В некоторых источниках сущностью называют группу однотипных объектов, а не отдельный объект. Отдельный же объект в группе называют экземпляром сущности. Знание этого факта, надеюсь, поможет вам не быть сбитым с толку.


    События тоже могут быть сущностями.

    В информационной системе, сущностями могут быть не только объекты реального мира, но и происходящие в этом мире события. Например, каждый факт поступления или продажи товара должен быть зарегистрирован в информационной системе. Каждый такой факт мы и рассматриваем как сущность и объединяем в группы «Поступление товара» или «Продажа товара».


    Один из способов выявления сущностей.




    Если при проектировании базы данных описать простыми предложениями то, что происходит, то большинство существительных и часть глаголов будут кандидатами на сущности.

    Например, у нас происходят события:
    1. Поставщики поставляют на нашу фирму товары.
    2. Сотрудники фирмы продают товары покупателям.

    В этом случае сущностями могут быть:
    1. поставщики,
    2. товары,
    3. поступление товара,
    4. сотрудники,
    5. покупатели,
    6. продажа товара.
  2. Атрибут - свойство сущности.

    Каждая сущность (каждый объект в сущности) может иметь много свойств. Эти свойства называют атрибутами. Например, человек имеет пол, имя, фамилию, рост, вес и так далее. Автомобиль имеет модель, цвет.

    Для каждого атрибута сущности, информацию о которой мы хотим хранить в БД, создается один столбец в таблице.

  3. Домен - множество допустимых значений, которые может принимать атрибут сущности (столбец таблицы).

    Мы уже рассматривали ограничение типом данных. И для хранения значения пола сотрудника (мужской или женский) нам нужен тип данных, хранящий семь символов – char(7). Семь символов позволяют ввести огромное количество слов. Но нам нужно позволить ввести ТОЛЬКО два возможных значения пола – мужской, женский.

    Еще пример, который упоминался выше – вес объектов реального мира не может быть отрицательным (даже равным нулю), а вот типа данных – положительное число нет. В данном случае доменом является все множество положительных чисел.

  4. Связь между сущностями - ссылки между таблицами. Очень часто атрибутом одной сущности является другая сущность. Например, в школе есть:
    1. сущность «ученики» ( таблица «pupils») – таблица всех детей, учащихся в этой школе. 
      
 pid |  pname  |   pburn    | pclass |      pheadteacher 
-----+---------+------------+--------+-------------------------------
   1 | Иванов  | 2010-03-25 |  3-А   | Стоянова Алла Борисовна
   2 | Петров  | 2011-08-14 |  3-Б   | Дашкевич Анна Ивановна
   3 | Сидоров | 2010-06-18 |  3-В   | Михайличенко Раиса Борисовна

    2. сущность «классы» (таблица «classes») - таблица всех классов в школе. 
      
 cnumber |         headteacher          
---------+------------------------------
 3-А     | Стоянова Алла Борисовна
 3-Б     | Дашкевич Анна Ивановна
 3-В     | Михайличенко Раиса Борисовна

    И в таблице «ученики» нам надо хранить данные о том, в каком классе ребенок сейчас учится и кто его классный руководитель. Мы должны осуществить связь между сущностями «pupils» и «classes». Для этого нам надо организовать ссылку из одной таблицы с полями другой.
